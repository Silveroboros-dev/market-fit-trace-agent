# Hackathon Asymmetry Check

Use this skill to decide what to build, cut, stub, or demo.

## Required output

1. Judge-visible "Aha!" moment
2. Features that directly support it
3. Backend work with low visible value
4. Features to cut
5. Features to stub
6. Demo-critical path
7. Highest-risk demo failure
8. Next three build tasks

## Rules

- Optimize for perceived magic backed by real implementation.
- Demo reliability beats architectural ambition.
- Cut anything that does not improve judge understanding, sponsor alignment, or credibility.
- Prefer replayable fixtures over fragile live dependencies.
